# Advanced Color Settings

## Why + UX stance

Some tenants insist on full control of their page's colors. But offering a color panel invites *everyone* to
tinker and waste time when the preset already looks good. So the panel is deliberately **buried at the bottom
and behind a toggle** ("Advanced Color Settings"), labeled "Changes override the preset theme." Most tenants
never open it; power users get complete control. Keep that friction on purpose.

## The model: preset defaults + compact token overrides (mirrors the Page Composer)

Presets already set the page's design tokens (`--sl-*`). Advanced Color Settings is simply a **per-token
override layer** that wins over the active preset — the exact same "defaults + compact overrides" shape as
`composition.overrides`:

- `page.theme.overrides = { "<token>": "<value>" }` — only deviations from the preset; empty = pure preset.
- Both renderers already consume `--sl-*` tokens, so applying overrides = emit the preset tokens, then a
  small override block re-setting the changed ones on the page root. Minimal render change; the live preview
  updates the same way.

Reset-to-preset = delete the override key (identical to how `toggleSection` clears a composition override).

## The key improvement over legacy: generate the panel from a token catalog

Legacy's shortcoming was structural: its Advanced panel was a **hand-built DOM form**
(`advancedColorOptions`, hardcoded field IDs in `stripe-cart/dist/dashboard/js/landing-pages.js` ~L2528–2600,
3910+) — so any element without a hand-added field was uncovered. The fix:

- A **canonical token catalog** where **every element/section declares its themeable tokens**
  (`{ token, label, group, kind: color | gradient }`). This is an output of the Builder Reframe (once every
  element is named, its tokens are named too).
- The Advanced panel is **generated by iterating the catalog** — grouped by element (Backgrounds, Text,
  Brand, CTA Button, Trust Badges, Savings, Featured, Avatar ring, Hero brand chip, …). Coverage becomes
  automatic and total: add an element → its tokens appear in Advanced Settings for free, with zero form work.

That is why this is **Phase 2** — it depends on the complete element/token catalog from the Builder Reframe.

## Gradients & composite tokens

Some tokens aren't a single color: the CTA is a **gradient (start + end)** → `--sl-cta-bg`
(`linear-gradient(...)`), matching the socialite work ([[SOCIALITE_PARITY]]). The catalog marks these
`kind: gradient` with two sub-inputs (as in the screenshots: CTA Button → Gradient Start / Gradient End).
The avatar ring is similar.

## Guardrails (optional, ties to the quality baseline)

Non-blocking **contrast warnings** (e.g. button text vs. button background, text vs. page background) surfaced
as nudges — never blocking publish. Reinforces the "quality baseline as warnings, not gates" principle from
[[LANDING_PAGE_GOAL_COMPOSITION]].

## Dependencies & phasing

- **Prerequisite**: Builder Reframe → complete element taxonomy + token catalog (each element names its
  themeable tokens).
- **Plan now; implement in Phase 2** (after the taxonomy is nailed).
- **Reuse**: the legacy panel's groups/labels are a fine starting point; we replace its hand-built form with
  the catalog-generated one and extend coverage to every token.

## Status

- **Shipped (v1):** the panel is **generated from a token catalog** in `composition_rules.json`
  (`token_catalog`: `{token: {label, group, kind}}`, grouped Backgrounds/Text/Brand/CTA/Badges/Price/…).
  Buried behind an "Advanced Color Settings" toggle; each token is a hex/CSS-color input with a live swatch,
  its placeholder showing the preset's effective value (read off the preview via `getComputedStyle`). Overrides
  persist as **`page.theme.tokens`** (the server already merges these over the preset — no backend change to
  apply them), restored on edit, auto-opening the panel when present. The preview applies overrides as inline
  `--preview-*` vars. Validation relaxed from hex-only to hex/rgb(a)/hsl(a). "Reset to preset" clears overrides.
  Adding a token to the catalog surfaces it automatically — no per-field form work (legacy's gap fixed).
- **Later:** gradients beyond the CTA; per-group reset; contrast warnings; auto-open on "custom" preset;
  more tokens as needed (the catalog is currently a curated ~19).

## Open decisions

- Token **granularity** per element (how fine-grained before it's overwhelming even for power users).
- **Reset UX**: per-token, per-group, and/or a global "reset to preset".
- Whether to expose **gradients beyond the CTA** (e.g. section backgrounds).
- Auto-enabling Advanced when a preset is "custom" (legacy did this) vs. always explicit.
